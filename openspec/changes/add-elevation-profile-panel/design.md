## Context

当前前端布局为左侧固定宽度侧边栏（440px） + 右侧弹性地图区域。路线规划完成后，距离、时间等数据仅在聊天消息中以文本卡片展示，高程数据（累计爬升、累计下降）同样混在文本中。用户无法直观感知路线地形特征。

后端 `elevation_service.py` 已具备高程查询能力（通过 open-elevation API），但返回的高程数据当前仅拼接在工具输出的文本末尾（"[高程数据]..."），未以结构化形式传递给前端。前端 `drawRoute()` 只绘制路线折线和起终点标记。

## Goals / Non-Goals

**Goals:**
- 将地图区域从全宽改为 3/4 宽，下方腾出空间放置路线数据面板
- 新面板展示：距离（km）、累计爬升（m）、累计下降（m）+ 海拔剖面图（填充面积图，模拟地形横截面）
- 后端返回结构化高程轨迹点数组（含累计距离），前端用 Canvas 渲染图表
- 基于高程数据计算 UCI 爬坡段（坡度 >= 3%，长度 >= 500m），在地图路线上叠加彩色标记
- 爬坡段侧边栏：右侧滑出，列出每个爬坡段的详情和分段海拔图
- 保持现有 earth-tone 设计系统一致

**Non-Goals:**
- 不引入第三方图表库（用原生 Canvas 绘制）
- 不修改百度地图 MCP 协议
- 不改变侧边栏（对话/路书区）功能

## Decisions

### 1. 布局方案：上地图 + 下面板 vs 左地图右面板

**选择**：上方地图（占 3/4 高度）+ 下方数据面板（占 1/4 高度）。

**理由**：海拔-距离图的横坐标为距离，与地图的水平维度不同，上下布局更自然。用户阅读流程图时自上而下看数据和图表。

### 2. 图表渲染：原生 Canvas vs 第三方库

**选择**：原生 Canvas。

**理由**：零依赖，包体积不增加；海拔剖面图采用纯填充面积图（无折线描边），仅靠渐变填充表现地形轮廓，Canvas 足够胜任；现有项目无构建工具，引入 npm 库需要额外配置。

### 3. 高程数据传递：扩展 SSE route 事件 vs 新增独立事件

**选择**：新增 `elevation` SSE 事件类型，在 `route` 事件之后发送。

**理由**：高程数据可能较大（每 100m 一个采样点，50km 路线约 500 个点），与路线数据分离有利于前端渐进渲染——先显示路线轨迹，再加载高程面板。route 事件保持现有结构不变，向后兼容。

### 4. 爬坡段识别算法

**选择**：在前端基于高程轨迹点计算坡度段。对每个采样点计算局部坡度（前向窗口差分），连续满足 UCI 标准（坡度>=3% 且长度>=500m）的区段合并为一个爬坡段。

**理由**：高程数据已在 `elevation` 事件中完整传递给前端，前端计算无需额外 API 调用；计算逻辑清晰，UCI 规则明确。

### 5. 爬坡段侧边栏交互

**选择**：右侧滑出面板（overlay 模式），通过切换按钮浏览不同爬坡段。

**理由**：不干扰地图和主面板布局；overlay 模式在不需要时可完全隐藏；切换设计直观——"第 X 个/共 Y 个" + 上下箭头。

## Risks / Trade-offs

- [大量轨迹点 Canvas 绘制性能] → 对高程点做前端降采样（最多绘制 300 个数据点）
- [open-elevation API 限流] → 已在后端做 100m 间隔采样，单次请求点数可控
- [爬坡段识别精度受采样间隔影响] → 注明 UCI 爬坡段为估计值，非官方认证
